# roles-and-flows.md — 角色與操作流程知識庫

> 用途：識別功能設計是否符合各角色的實際操作路徑與系統權限。

---

## 角色總覽

| 角色 | 中文 | 主要職責 | 使用系統 |
|------|------|---------|---------|
| Sales | 業務/門市人員 | 招攬客戶、開單、變更訂單 | 訂單系統、報價系統 |
| OP | 操作人員 | 出發日管理、供應商串接、行程確認 | OP 作業系統、供應商系統 |
| Tour Leader | 領隊 | 帶團、現場協調、事後報告 | 領隊 App（若有）、LINE 群組 |
| Customer | 客戶 | 報名、付款、查詢訂單、提出需求 | 官網、客服 |
| Finance | 財務 | 帳務核銷、退款作業 | 財務系統、訂單系統 |

---

## 業務（Sales）操作路徑

**主流程**
1. 客戶詢問 → 業務報價 → 客戶確認 → 業務開單（訂單系統）
2. 開單時填入：出發日、人數、房型需求、特殊需求
3. 收取訂金（部分付款）→ 系統狀態變更為「已訂金」
4. 尾款截止前通知客戶付清 → 系統狀態變更為「已付清」

**常見業務操作場景**
- 客戶要求「先保留名額，過幾天再付訂金」→ 系統是否支援「暫保留」？時間限制？
- 客戶要求加人（同訂單新增成員）→ 名額是否還有？費用如何重算？
- 客戶要求換出發日 → 原出發日名額釋放、新出發日名額扣減，訂單歷史如何保留？
- 業務直接修改訂單 vs 需要主管審核的情境（高金額退款、特殊折扣）

**業務看不到的事**
- 供應商實際確認狀態（飯店/航空是否真的有位）
- OP 的後台備註（業務端不一定即時同步）

## OP（操作人員）操作路徑

**主流程**
1. 收到確認成行的出發日 → 開始向供應商確認（飯店、地接、航班）
2. 建立「行程確認書」→ 送給客戶
3. 出發前 N 天：整理名單（含護照資料）→ 送件給地接社 / 航空公司
4. 出發當天：交接給領隊
5. 結團後：核對費用、輸入帳單

**OP 的高壓力場景**
- 同一時間多個出發日同時進入確認程序 → 資料整理量大
- 客戶臨時提供護照資料 → OP 需手動補件，容易出錯
- 飯店確認回覆慢 → OP 需要追蹤狀態，系統若無提醒會遺漏
- 出發前有人退出 → OP 需通知飯店、地接社調整人數，並確認費用影響

**OP 需要但常常缺少的資訊**
- 訂單上的「特殊需求備註」是否完整傳到供應商端
- 客戶的文件（護照、保險）是否齊全的狀態追蹤

## 領隊（Tour Leader）操作路徑

**出發前**
- 收到 OP 整理的行程確認書與名單
- 了解成員特殊需求（素食、輪椅、醫療需求）

**帶團中**
- 現場點名與管理
- 突發狀況處理（班機延誤、成員生病、行程異動）
- 與地接社溝通

**結團後**
- 撰寫領隊報告（回報異常、建議改善）
- 回報帳務（現場支出、小費收取）

**常見痛點**
- 領隊取得名單的方式通常是 PDF 或紙本，難以即時更新
- 帶團中的異動（例如一人提早回國）需要通知多個後端人員，流程不清晰

## 客戶（Customer）操作路徑

**報名流程**
1. 瀏覽行程 → 選擇出發日 → 填寫報名資料（含同行成員）
2. 線上付款（訂金 or 全額）
3. 收到確認信
4. 收到行程確認書（出發前 N 天）
5. 出發

**客戶常見痛點**
- 多人同行時，需要填寫所有成員資料，表單體驗差
- 付款後不確定「訂單是否真的確認」（候補狀態不清楚）
- 行程異動通知不即時（有時只靠業務電話）

**客戶不理解的事**
- 「候補」的意思（很多客戶以為付了款就確定出發）
- 取消費用的計算方式（出發前幾天取消扣幾%）

## 角色間的常見摩擦點

| 摩擦點 | 涉及角色 | 風險 |
|--------|---------|------|
| 業務承諾了 OP 無法執行的服務 | Sales vs OP | 客戶期望落差 |
| 客戶提需求給業務但 OP 不知道 | Customer → Sales → OP | 資訊斷層 |
| 領隊名單是出發前才給，無法提前準備 | OP vs Tour Leader | 現場混亂 |
| 退款由財務處理但客戶問業務 | Finance vs Sales | 業務給錯資訊 |
| 候補狀態客戶問業務但業務也不確定 | Sales vs System | 客服壓力 |
