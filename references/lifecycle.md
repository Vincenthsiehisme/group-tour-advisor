# lifecycle.md — 團旅生命週期與角色分工知識庫

> 用途：讓 skill 能識別 PM 當前討論的是哪個生命週期階段，並對應正確的風險知識層。
> 每個階段都標明：核心產出、主要角色、高發風險點、與其他階段的銜接依賴。

---

## 生命週期總覽

```
[P1 產品企劃] → [P2 業務招攬] → [P3 訂團與團控] → [P4 出團執行] → [P5 結團與檢討]
      ↑                                                                      ↓
      └──────────────────── 回寫優化（標準化循環）────────────────────────────┘
```

## P1｜產品企劃

**核心任務**：決定「這條線能不能賣、怎麼賣」

| 項目 | 說明 |
|------|------|
| 主要活動 | 市場調查、行程設計、成本估算、產品內容訓練、定價與毛利結構確認 |
| 主要角色 | 產品/企劃（主導）、業務（市場回饋）、財務（成本審核） |
| 核心產出 | 行程草案、報價單、產品訓練手冊、售價與毛利表 |

**高發風險點**
- 成本估算未包含「淡旺季差異」→ 同一行程不同出發日毛利落差極大
- 行程設計未考量「最低成團人數的盈虧平衡點」→ 低於門檻仍強行出發則虧損
- 產品內容訓練（業務話術、賣點）與實際行程內容有落差 → 客戶期望管理失敗

**銜接下一階段的關鍵資訊**
- 每個出發日的 min pax / max pax / 售價 / 截止報名日
- 取消政策與退款規則
- 哪些供應商需要提前鎖位（航班 block seat、特殊飯店 allotment）

**功能設計常見涉及此階段的關鍵字**
「行程管理」「產品上架」「定價設定」「毛利試算」「出發日建立」「產品審核」

## P2｜業務招攬

**核心任務**：接單、需求確認、報價、識別決策結構

| 項目 | 說明 |
|------|------|
| 主要活動 | 門市接洽、同業配合、線上通路導流、企業客戶洽談、需求確認、報價作業 |
| 主要角色 | 業務（主導）、產品/企劃（報價支援）、客戶/企業採購 |
| 核心產出 | 詢價記錄、報價單、初步需求確認單 |

**三角決策結構（團體業務特有）**

> 「誰是決策者、誰是付款者、誰是參加者」三者不一定是同一人。

| 角色 | 說明 | 常見情境 |
|------|------|---------|
| 決策者 | 決定是否成行 | 公司主管、家族長輩、工會幹部 |
| 付款者 | 實際付款的人/單位 | 公司財務、主辦人、個人 |
| 參加者 | 實際出發的人 | 員工、家庭成員、工會成員 |

**高發風險點**
- 決策者與參加者不同 → 行程異動需知會兩方，通知流程設計容易遺漏
- 企業團的付款者是公司帳戶 → 發票、收據、合約需求與個人客戶不同
- 同業配合（批發 / 拼團）→ 上下游旅行社之間的分潤與責任歸屬需在訂單中體現

**銜接下一階段的關鍵資訊**
- 確認人數（成人/兒童/嬰兒）與房型需求
- 特殊需求（素食、輪椅、醫療、簽證國籍）
- 付款方式與發票抬頭

**功能設計常見涉及此階段的關鍵字**
「詢價」「報價」「開單」「需求確認」「訂單建立」「企業客戶」「同業配合」「拼團」

## P3｜訂團與團控

**核心任務**：控位、控房、訂票、簽證、保險、地接，依成團狀況調整名單

| 項目 | 說明 |
|------|------|
| 主要活動 | 航班訂位、飯店 allotment 確認、簽證申辦、保險投保、地接社確認、名單管理、成團狀態追蹤 |
| 主要角色 | OP/計調/團控（主導）、票務、簽證專員、業務（名單異動通知） |
| 核心產出 | 確認書、名單、機票票號、飯店確認單、行程確認書 |

**這是最容易出問題的階段，直接連動庫存與成本**

**高發風險點**
- 名單異動（加人/退出）未及時通知供應商 → 飯店房間數對不上、航班改名費用
- 多渠道同時扣配額 → 超賣（見 `business-rules.md`）
- cut-off date 管理失誤 → 過期後才想確認，供應商已釋放名額
- 簽證材料不齊 → 卡在申辦，影響出發
- 成團狀態（候補/確認/取消）與業務端資訊不同步 → 業務仍繼續招攬已滿團

**名單管理的核心複雜度**
- 同一訂單可能多次異動（加人、換人、退出）`[業界通例]`
- 每次異動都可能觸發：配額重算、費用重算、房型重配、供應商通知 `[雄獅規則－待確認：實際上哪些異動會自動觸發哪些重算]`
- 部分成員退出後，剩餘成員的房型可能需要重新分配（例如雙人房變單人房加收差價）`[雄獅規則－待確認：是否有標準 SOP 或由業務人工判斷]`

**銜接下一階段的關鍵資訊**
- 最終名單（護照資料齊全）
- 票號與確認書
- 特殊需求是否已通知地接

**功能設計常見涉及此階段的關鍵字**
「控位」「控房」「名單管理」「成團確認」「候補」「配額」「cut-off」「訂票」「簽證」「保險」「地接確認」「行程確認書」

## P4｜出團執行

**核心任務**：交付旅遊體驗，現場品質管理

| 項目 | 說明 |
|------|------|
| 主要活動 | 行前說明會、出發報到、帶團、食宿交通安排、緊急應變 |
| 主要角色 | 領隊/導遊（主導）、地接社、OP（後援）、客服（緊急支援） |
| 核心產出 | 行前說明資料、領隊日誌、緊急事件紀錄 |

**這段的核心是交付品質，客人體驗主要在這裡形成**

**高發風險點**
- 領隊拿到的名單是靜態 PDF，無法即時反映最後一刻的名單異動
- 成員特殊需求（素食、輪椅）沒有傳到地接 → 現場才發現
- 緊急狀況（班機延誤、成員生病）的通報流程不清晰，OP 與客服不知道該誰處理
- 行程現場異動後，後台訂單無對應記錄 → 結團帳務無法對齊

**系統支援缺口**（常見）
- 領隊端通常沒有系統工具，靠 LINE 群組與紙本運作 `[雄獅規則－待確認：目前是否有領隊 App 或數位工具]`
- 現場緊急支出無法即時記帳，事後補填容易出錯 `[雄獅規則－待確認：現行帳務補填流程]`

**銜接下一階段的關鍵資訊**
- 領隊日誌（行程異動、緊急事件）
- 現場帳務（自費行程金流、小費）
- 客訴事件紀錄

**功能設計常見涉及此階段的關鍵字**
「行前說明」「出發報到」「領隊 App」「現場通報」「緊急應變」「帶團管理」

## P5｜結團與檢討

**核心任務**：回收回饋、處理客訴、結算供應商、回寫下一輪產品企劃

| 項目 | 說明 |
|------|------|
| 主要活動 | 滿意度問卷、客訴處理、供應商帳款核銷、實際成本 vs 預算差異分析、產品優化建議 |
| 主要角色 | 客服（客訴）、財務（帳款）、產品/企劃（回寫優化）、OP（供應商帳務） |
| 核心產出 | 滿意度報告、客訴記錄、供應商結算單、產品優化建議書 |

**做好這階段，產品才能逐步標準化與優化**

**高發風險點**
- 客訴沒有系統化記錄 → 同樣問題重複發生
- 實際成本與預算差異沒有回寫到下一輪報價 → 毛利持續被侵蝕
- 供應商帳款核銷靠手工對帳 → 容易漏掉或多付

**回寫循環的關鍵**
- 出現頻率高的客訴 → 應回寫到 P1 的行程設計或產品說明
- 成本結構偏差 → 應回寫到 P1 的定價模型
- 名單管理問題 → 應回寫到 P3 的 OP 作業流程

**功能設計常見涉及此階段的關鍵字**
「滿意度調查」「客訴管理」「結算」「帳款核銷」「產品優化」「回饋收集」
