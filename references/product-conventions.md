# product-conventions.md — 旅遊產品設計慣例知識庫

> 用途：確認功能設計是否符合旅遊產品的 UI/UX 特殊需求，避免套用一般電商邏輯導致的體驗問題。

---

## 1. 多人同行訂單（Group Booking）

**特殊性**：旅遊訂單常見「一人代訂，多人同行」的情境，與一般電商的「一人一訂單」完全不同。

**設計要點**
- 「主訂人」vs「同行成員」需分開資料欄位：主訂人負責付款與溝通，成員只需填旅遊所需資料（護照等）
- 成員資料填寫時機：可允許「先報名、後補件」，但需有截止日與提醒機制
- 兒童與嬰兒的欄位需求不同：嬰兒通常不佔座位，但需填年齡確認費率

**常見設計錯誤**
- 把所有成員當主訂人要求填完整資料 → 報名流程太長，放棄率高
- 沒有「補件提醒」機制 → OP 出發前才發現名單不完整
- 成員數量變動（加人/減人）後，訂單總金額沒有即時更新

## 2. 出發日選擇（Departure Date Picker）

**特殊性**：旅遊的「出發日」不是日曆任意選，而是固定班次，且每個班次有不同的剩餘名額與狀態。

**設計要點**
- 出發日選擇器應顯示：日期、剩餘名額、候補狀態、價格（可能因日期不同）
- 「候補」狀態需有明確說明，避免客戶誤以為已確認
- 過去的出發日不應顯示（或灰化），截止報名的出發日也需標示

**常見設計錯誤**
- 用一般日曆元件，沒有整合剩餘名額資訊
- 候補狀態只用文字標示，客戶不理解意思
- 出發日切換後，價格沒有跟著更新

---

## 3. 房型選擇（Room Type Selection）

**特殊性**：團旅的房型選擇邏輯與飯店訂房不同，依人數組合有固定規則。

**常見房型組合**
- 雙人房（兩大床 or 雙人床）
- 單人房（加收單人費）
- 三人房（加床，可能有品質差異）
- 子母房（帶兒童的家庭常見）

**設計要點**
- 當同行人數為奇數時，需主動提示「單人加收費」並讓客戶確認
- 房型不一定是「選」的，而是「依人數自動分配」，但客戶有偏好時需有備註欄
- 同一出發日的房型配額可能各自獨立（雙人房只剩 2 間）

**常見設計錯誤**
- 只讓客戶「選房型」但不告知單人費
- 三人房的品質說明不清楚（客戶期望落差）
- 沒有「備註房型偏好」的欄位

## 4. 行程異動通知（Itinerary Change Notification）

**特殊性**：團旅行程在出發前可能異動（景點取消、飯店更換、航班時間調整），客戶需要即時知道。

**設計要點**
- 通知層級：微幅異動（飯店等級不變但換了飯店）vs 重大異動（行程天數變更）需分開處理
- 通知時機：異動確認後立即通知，不等到出發前統一告知
- 客戶回應機制：重大異動是否允許客戶免費取消？系統需支援「異動觸發的特殊取消流程」

**常見設計錯誤**
- 只有 email 通知，沒有 App push / SMS → 開信率低
- 通知只說「行程有異動」但沒有說明具體變動內容
- 沒有設計「客戶確認收到」的機制

---

## 5. 文件收集（Document Collection）

**特殊性**：出國旅遊需要收集護照、簽證等文件，這是一般電商沒有的需求。

**設計要點**
- 護照資料欄位：姓名（英文）、護照號碼、有效期限、國籍
- 需有「截止日期」與進度提醒（PM 這邊缺幾份）
- 允許客戶更新資料（護照即將到期需換發）

**常見設計錯誤**
- 只在報名時收集，沒有更新機制
- OP 無法一眼看出「哪些成員還沒交文件」
- 護照英文姓名與護照格式不符（例：空格、縮寫問題）→ 航空報名錯誤

## 6. 彈性座位 / 加購項目（Optional Add-ons）

**常見加購項目**
- 自費行程（特定景點的額外活動）
- 座位升等（飛機商務艙）
- 旅遊保險
- 接送服務

**設計要點**
- 加購項目的「截止時間」可能早於出發日（例如：保險需出發前 3 天前購買）
- 部分加購有名額限制（自費行程的車位）
- 加購退款規則可能與主行程不同

**常見設計錯誤**
- 加購的截止日期沒有在介面上明確標示
- 退款流程把加購與主行程混在一起計算
- 自費行程的人數未與主行程名單連動

## 設計慣例快速檢查清單

設計旅遊相關功能前，確認以下問題：

- [ ] 是否考慮「多人同行」的代訂情境？
- [ ] 出發日的狀態（候補/確認/截止）是否在 UI 上清楚呈現？
- [ ] 人數奇偶數是否觸發單人費提示？
- [ ] 取消流程是否有「全部取消」vs「部分成員取消」的分支？
- [ ] 文件收集是否有截止日與進度追蹤？
- [ ] 行程異動是否有通知機制，且支援重大異動的特殊取消流程？
- [ ] 加購項目的截止日與退款規則是否與主行程分開？

## 帛琉行程的飯店選配邏輯（雙維度）

> 來源：雄獅銷售實務教材 2017年1版

帛琉行程選飯店時，業務和 OP 參考兩個獨立維度，產品系統若要支援選房或比較，建議同時呈現：

**維度一：距出海碼頭距離**（影響每天出海活動的集合體驗）
- 碼頭旁（最近）→ 約5分鐘 → 約10分鐘（多間） → 約15–25分鐘 → 約30分鐘（最遠）

**維度二：價位高低**
- 老爺（五星，唯一有私人碼頭）> 大飯店（四星，房間數最多）> 愛來水漾（四星，唯一有水上樂園）

**設計建議**：
- 若行程頁面有飯店選配功能，應同時標示「距碼頭距離」與「星等/價位」
- 老爺飯店有私人碼頭，可從飯店直接出海；其他飯店需集合後統一前往，集合時間差異需在行程說明中呈現
- 「帛琉唯一三人房」為老爺飯店 → 若旅客為三人組合，業務需主動說明選房限制 [雄獅規則－待確認]

## Wi-Fi 申請的行前提醒時機

> 適用：美加線、關帛線（視行程附帶 Wi-Fi 方案）

**業務慣例**：Wi-Fi 申請有截止期限，需在出發前特定天數通知旅客 [雄獅規則－待確認，具體天數需確認]

**設計建議**：
- Wi-Fi 申請提醒應由系統自動觸發（如出發前 14 天發通知），不應依賴業務人工逐一提醒
- 若 Wi-Fi 申請截止日與護照/文件截止日不同，需分開顯示，避免旅客混淆
